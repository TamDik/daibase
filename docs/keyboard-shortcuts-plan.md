# キーボードショートカット実装計画

## 目的

Daibase の主要操作をキーボードから実行できるようにし、ユーザーが割り当てを変更、解除、初期値へ復元できる仕組みを提供します。

組み込み機能だけを特別扱いする設計にはせず、将来はプラグインが宣言したコマンドも同じコマンド一覧、設定画面、競合検査、実行経路へ登録できる構成を目指します。

## 現在の実装状況

### 対応済みのコマンド

| コマンド ID          | 操作                     | デフォルト       |
| -------------------- | ------------------------ | ---------------- |
| `search.global`      | 全体検索                 | `Mod+K`          |
| `search.page`        | ページ内検索             | `Mod+F`          |
| `navigation.back`    | 戻る                     | `Alt+ArrowLeft`  |
| `navigation.forward` | 進む                     | `Alt+ArrowRight` |
| `shortcuts.open`     | ショートカット一覧を開く | `Mod+Shift+K`    |

`Mod` は実行時に macOS の Command または Windows / Linux の Control と照合します。保存値には OS 固有のキー名を使わず、共通表現を保持します。

### コマンド定義

フロントエンドの `src/lib/keyboardShortcuts.ts` に、次の情報を持つ `ShortcutCommand` を定義しています。

```ts
type ShortcutCommand = {
  id: string;
  title: string;
  description: string;
  category: string;
  defaultBinding: string;
  source: "builtin" | `plugin:${string}`;
};
```

- 設定値はコマンド ID をキーにして保持します。
- キーボードイベントは `Mod+Shift+K` のような共通文字列表現へ正規化します。
- 同じ割り当てを持つ別コマンドがある場合は設定を拒否します。
- 入力欄、テキストエリア、選択欄、編集可能要素では、修飾キーを伴わないショートカットを実行しません。

### 設定の永続化

現在は `localStorage` の `daibase.keyboard-shortcuts.v1` に、コマンド ID と割り当ての対応を JSON で保存します。

- 初回起動時はデフォルト値を使用します。
- 項目ごとに割り当てを解除できます。
- 項目ごとにデフォルトへ戻せます。
- 全項目を一括でデフォルトへ戻せます。
- 保存値が壊れている場合はデフォルト値へフォールバックします。
- 新しい組み込みコマンドが追加された場合、保存値に存在しない項目にはデフォルト値を補います。

### 実行経路

`HomePage` がグローバルな `keydown` を受け取り、割り当てに一致するコマンド ID を解決します。現在はコマンド ID ごとの分岐から既存の画面操作を呼び出します。

```text
keydown
  -> キー表現を正規化
  -> ユーザー設定からコマンド ID を解決
  -> コマンド ID に対応する画面操作を実行
```

全体検索とページ内検索は request ID を更新し、既存コンポーネントの表示処理を呼び出します。戻る、進む、ショートカット一覧は既存のナビゲーション処理を再利用します。

コマンドメタデータは `src/lib/commandRegistry.ts` の共通レジストリへ移し、ショートカット、コマンドランチャー、コマンド一覧が同じ定義を参照します。コマンドランチャーでは `>` から始まる入力をコマンドモードとして扱い、曖昧検索、矢印選択、Tab 補完、Enter 実行に対応しています。

### 設定画面

`Special:Shortcuts` をグローバルな Special ページとして実装しています。Special ロケーションの解決は Rust 側を正とし、React 側は返された表示種別を描画します。

画面では次を表示、操作できます。

- コマンドの機能アイコン、表示名、説明、カテゴリ、コマンド ID。
- `⌘`, `⌥`, `⇧`, 矢印などのキートップ表現。
- キー入力による割り当て変更。
- 入力欄右端のクリアボタンによる割り当て解除。
- 個別のデフォルト復元と全体のデフォルト復元。
- 重複割り当て時のエラー表示。

### コマンド一覧

`Special:Commands` をグローバルな Special ページとして実装しています。組み込みコマンドの名前、説明、カテゴリ、コマンド ID、提供元、現在のショートカットを表示し、その場で実行できます。

コマンド型の `source` は `builtin` と `plugin:{pluginId}` を区別できるため、将来 Plugin Host が検証したコマンドを同じレジストリへ追加できます。

## 現在の責務分担

### Rust

- `Special:Shortcuts` のロケーション解決。
- `OpenLocationResult::SpecialShortcuts` の返却。
- Special ページ一覧への登録。

### React / TypeScript

- コマンド定義とデフォルト割り当て。
- キーイベントの正規化、競合検査、設定の読み書き。
- コマンド実行と画面状態の更新。
- 設定画面の描画とユーザー操作。

ショートカット設定はアプリ UI の設定であり、namespace コンテンツではないため、現時点では Rust のコンテンツ管理や MCP 編集ツールの対象にはしません。

## 今後の実装計画

### 1. コマンドレジストリと実行処理の分離

コマンドメタデータの共通レジストリ化と、ランチャー、ショートカット、一覧画面から同じ実行関数を呼ぶ経路は実装済みです。次に、現在 `HomePage` に残っているコマンド ID ごとの実行分岐を、実行関数を登録できるレジストリへ移します。

```ts
type RegisteredCommand = ShortcutCommand & {
  execute: (context: CommandContext) => void | Promise<void>;
  isEnabled?: (context: CommandContext) => boolean;
};
```

目標:

- コマンド定義、表示、割り当て、実行を同じ ID で接続する。
- 画面コンポーネントが巨大な ID 分岐を持たないようにする。
- メニュー、コマンドランチャー、ショートカットから同じ実行処理を呼べるようにする。

### 2. コンテキスト条件

ページ表示中だけ使える操作などを、実行時のコンテキストで判定できるようにします。

候補:

- `activeNamespace` の有無。
- Page / File / Special ページの表示種別。
- 閲覧モード、編集モード。
- モーダルや入力フォーカスの状態。
- プラグインの有効状態と permission。

無効なコマンドはイベントを消費せず、設定画面では必要に応じて利用条件を表示します。

### 3. プラグインコマンド contribution

プラグイン manifest に `command` contribution を追加し、Plugin Host が検証した定義だけをコマンドレジストリへ登録します。

想定例:

```json
{
  "kind": "command",
  "id": "open-calendar",
  "name": "Open Calendar",
  "description": "カレンダーページを開きます。",
  "category": "Calendar",
  "defaultBinding": "Mod+Shift+C"
}
```

アプリ内の完全なコマンド ID はプラグイン ID を含む形式にします。

```text
plugin:com.example.calendar/open-calendar
```

要件:

- manifest 読み込み時に ID、文字数、デフォルトキーを検証する。
- 組み込みコマンドとプラグインコマンドを同じ一覧へ表示する。
- プラグイン無効化、削除時にコマンドを実行不能にする。
- Plugin Host 経由で実行し、DOM や Tauri command を直接操作させない。
- Host API の permission と現在の context を実行時にも検査する。

この contribution を導入する際は `docs/plugin-development.md` と `docs/plugin-host-design.md` も更新します。

### 4. 設定ストレージの拡張

プラグインコマンド対応前に、設定ストレージを次の要件へ拡張します。

- 一時的に未登録のコマンド ID も保存値から失わない。
- プラグインを再有効化したときに以前の割り当てを復元する。
- schema version と migration を明示する。
- 複数ウィンドウで設定変更を同期する。
- 必要になった段階で `localStorage` から Rust 管理のアプリ設定へ移行する。

namespace ごとの設定ではなく、原則としてアプリユーザー単位の設定として扱います。将来 namespace ごとの上書きが必要になった場合も、グローバル設定とは別レイヤーにします。

### 5. OS 別表示とキー仕様

現在、保存と照合は OS 非依存ですが、キートップの `Mod` 表示は `⌘` 固定です。Windows / Linux では `Ctrl` または `⌃` 相当へ切り替える表示層を追加します。

追加検討:

- OS 予約ショートカットとの競合警告。
- `Ctrl` と `Meta` を明示的に区別する割り当て。
- ファンクションキー、テンキー、記号キーの正規化。
- キーボード配列に依存する `event.key` と物理位置を表す `event.code` の使い分け。
- 連続キーやコード形式のショートカットは、単一キー方式の安定後に別仕様として検討する。

## テスト方針

現行テストでは次を確認しています。

- デフォルト割り当ての生成。
- キーボードイベントの正規化。
- 重複割り当ての検出。
- 全体検索、ページ内検索、設定画面のショートカット実行。
- 割り当て変更後の新しいキーによる実行。
- キートップ表示。
- 割り当て解除、個別デフォルト復元。
- `Special:Shortcuts` の Rust ロケーション解決。

コマンドレジストリやプラグイン contribution を追加するときは、次を追加します。

- 組み込みコマンドとプラグインコマンドの登録、解除、ID 競合。
- plugin permission と context による実行許可、拒否。
- 無効化、削除、再有効化後の設定保持。
- schema migration と壊れた設定からの復旧。
- macOS、Windows、Linux の表示ラベルとイベント照合。

## 完了条件

現行の組み込みショートカット機能は実装済みです。将来のプラグイン対応は、次を満たした時点で完了とします。

- command contribution の schema と Plugin Host 実行 API が定義されている。
- 組み込み、プラグインを問わず、同じレジストリから一覧表示、競合検査、実行ができる。
- プラグインの無効化や削除で不正な実行経路が残らない。
- ユーザー設定がプラグインの一時的な未登録をまたいで保持される。
- OS ごとの主要なキー表記とイベント照合がテストされている。
